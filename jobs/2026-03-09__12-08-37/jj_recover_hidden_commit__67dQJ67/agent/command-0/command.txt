'# Recovering a Hidden (Abandoned) Commit in Jujutsu

## Background

As an OS engineer comparing `jj` to Git, you are exploring how Jujutsu handles abandoned commits that are no longer visible in the default log. In Git, abandoning a branch leaves commits effectively unreachable — they live as "dangling commits" in the object store, discoverable only via `git reflog` or `git fsck --unreachable`. In Jujutsu, the analogous concept is a **hidden** commit. When you run `jj abandon <rev>`, the commit is not deleted — it is hidden from the default log view but still stored. Unlike Git'"'"'s reflog, Jujutsu provides a first-class **operation log** (`jj op log`) that records every mutation. Using the operation log, you can rediscover commits that were abandoned in earlier operations.

## Starting Environment

You have a jj repository at `/home/user/kernel-patch`. The repository has the following **visible** commit history:

```

o  init: add initial Makefile              (contains file: Makefile)
```

However, there is also a **hidden** commit in the repository. This commit was abandoned by a previous engineer who thought the work was no longer needed. The hidden commit:
- Has description: `feat: add watchdog timer support`
- Contains a file named `watchdog.c` with content `/* watchdog timer implementation */`
- Is a child of the root commit (it was on a separate line of history)

This hidden commit does **not** appear in `jj log` (the default visible history). Your mission is to find it and restore it.

## Your Task

Complete the following steps to recover the hidden commit.

### Step 1: Discover the hidden commit using the operation log

Jujutsu'"'"'s operation log records every mutation to the repository. You can use the `at_operation()` revset function to look at what commits were visible at a previous operation. Run:

```bash
jj log -r '"'"'at_operation(@-, all())'"'"'
```

This shows all commits (including now-hidden ones) that were visible at the immediately previous operation (`@-` means the operation before the current one). You should see the commit with description `feat: add watchdog timer support` with a `(hidden)` marker. Note its **commit ID** (the hex SHA shown on the right in the log).

Alternatively, you can look at the full operation log:

```bash

```

And inspect a specific earlier state:

```bash

```

### Step 2: Restore the hidden commit into visible history

Once you have the **commit ID** (not the change ID) of the hidden commit, use `jj new` to create a new visible commit whose parent is the hidden commit. This brings the hidden commit'"'"'s ancestry into the visible graph.

```bash

```

Note: For hidden commits, you must use the full **commit ID** (a hex SHA) rather than the change ID (the short alphanumeric identifier). The commit ID is visible in the `at_operation` log output.

After running this command, the hidden commit becomes visible in `jj log` because the new working-copy commit'"'"'s parent chain now references it.

## Key jj Concepts Demonstrated

- **Hidden commits**: `jj abandon` makes commits hidden (not deleted)
- **Operation log** (`jj op log`): every jj mutation is recorded; unlike Git'"'"'s reflog, it is first-class and structured
- **`at_operation(op, revset)`**: evaluates a revset against an earlier operation'"'"'s state, surfacing commits that were visible then but are hidden now
- **`jj new <rev>`**: creates a new change on top of any commit, even hidden ones — making the hidden commit visible as an ancestor
- **`@-` in operation revset context**: the operation immediately before the current one
