## Background

`jj` uses `.gitignore` files to determine which files should not be automatically tracked. This works the same way as in Git — pattern rules in `.gitignore` tell jj which paths to ignore.

However, there is an important nuance: **files that are already being tracked will remain tracked even if they match an ignore pattern**. This means that adding an entry to `.gitignore` is not enough by itself to remove a file from the current working-copy commit. You must explicitly stop tracking such files using the `jj file untrack` command.

This is a common tripping point for developers migrating from Git, where `git rm --cached <file>` serves a similar purpose.

## Key Concepts

### Why doesn'"'"'t adding .gitignore automatically untrack files?

In jj (and Git), ignore rules only prevent *new* automatic tracking of files. Files that were already part of a commit remain tracked until you explicitly remove them. The workflow is:

1. Add the ignore pattern to `.gitignore`
2. Explicitly untrack already-tracked files with `jj file untrack <paths>`

### Command option tradeoffs

- `jj file untrack <path>` — stops tracking a specific path; **requires** the path to be ignored first
- `jj restore --from <rev> <path>` — restores file content from another revision; does not change tracking
- `jj abandon` — abandons a revision entirely; not suitable for removing individual files from tracking

Understanding which command to use in each context is an important part of working effectively with jj.
